# Soft Stock — ไม่ตัดสต๊อกแบบ Atomic

WARROOM Customer Gateway เป็นระบบสั่ง Merchandise ที่**ไม่จัดการการชำระเงินภายในระบบ** (ทีมจะติดต่อ Member กลับเพื่อเก็บเงินผ่าน PromptPay) จึงไม่มีจังหวะที่จะ commit แบบ atomic ได้จริง ที่จะใช้เป็นจุดตัดสต๊อกแบบเด็ดขาด

## การตัดสินใจ

ระบบเก็บค่า `stock_remaining` ของแต่ละ Variant ไว้เป็น**ตัวเลขบอกใบ้ให้ Member เห็น**เท่านั้น (หน้าจอแสดงเป็น "พร้อมส่ง" / "เหลือน้อย" / "สินค้าหมดชั่วคราว") ระบบ**ไม่บล็อก**การสร้าง Order เมื่อสต๊อกเป็น 0 และ**ไม่หักสต๊อก**อัตโนมัติเมื่อ Member สั่งสินค้า การปรับสต๊อกทำโดย Admin ด้วยมือ และทุกครั้งจะถูกบันทึกใน Audit Log

## เหตุผล

1. **ลดความซับซ้อนเรื่อง concurrency อย่างมาก** — ไม่ต้องใช้ row-level lock, optimistic concurrency หรือ SAGA pattern
2. **ความเสี่ยงขายเกินถูกลดด้วย flow ติดต่อกลับ** — ทีมจะตรวจสต๊อกจริงก่อนยืนยันทุกครั้ง หากของหมดจริงก็สามารถปฏิเสธ Order นั้นได้
3. **แหล่งข้อมูลความจริงของสต๊อกอยู่ที่คลังจริงของทีม ไม่ใช่ในฐานข้อมูล** — ต่อให้ตัดสต๊อกแบบ atomic ในฐานข้อมูล ก็ไม่ได้ atomic จริงในชีวิตจริงอยู่ดี (ของอาจเสียหาย, นับผิด, ฯลฯ)

## ผลที่ตามมา

- ผู้อ่านโค้ดที่คาดหวังรูปแบบ e-commerce ทั่วไป จะรู้สึกแปลกที่ไม่มี `select for update` หรือ logic จองสต๊อก — **ห้าม refactor ให้เป็น atomic** จนกว่าจะมีระบบการชำระเงินภายในระบบ
- หน้าจอแสดงสต๊อกจะเป็นแบบ eventually-consistent — Member 2 คนเห็น "เหลือน้อย" พร้อมกันและสั่งพร้อมกันได้ ทีมเป็นผู้จัดการตอนติดต่อกลับ
- หากในอนาคตเพิ่มการชำระเงินภายในระบบ ต้องทบทวนการตัดสินใจนี้ใหม่ (อาจถูกแทนที่ด้วย ADR ฉบับใหม่)
